# 我的想法：为什么做这个项目

这个项目最初不是为了写一个漂亮的工具，而是为了应对一个非常具体的现实问题：房源开放窗口短、人工操作链路长、登录态容易失效、多账号信息重复录入成本高。人在开放时间点附近要同时盯页面、刷新、选择房源、填写资料、处理滑块和网络波动，任何一个环节慢几秒都可能错过。

所以项目的第一目标是把“重复、机械、容易出错”的部分交给脚本：

- 账号资料只录入一次。
- Cookie 和 token 统一管理。
- 房源目标可以提前配置。
- 开放前自动预热，开放后自动提交。
- 失败原因实时展示，不靠猜。

## 从人工抢到系统化执行

人工抢房的核心矛盾是信息流太多：页面状态、登录态、验证码、房源开放时间、接口响应、网络延迟都混在一起。脚本的价值不只是更快，而是把这些因素拆开：

- 认证状态由 Cookie/token 承担。
- 安全校验由 CSRF、WAF Cookie 和 Referer 承担。
- 提交内容由配置文件承载。
- 节奏控制由调频参数承载。
- 成功/失败判断由响应分类承担。

一旦拆开，系统就可以做到可观察、可复盘、可调整。

## 为什么有两个子项目

南山和青年驿站看起来都是“抢房”，但技术形态不同：

- 南山基于 `sz.inboyu.com` 的活动页和表单提交接口，需要处理 Cookie、CSRF、WAF、代理、活动 ID、房源项目 ID等。
- 青年驿站基于 `api-home.szyouth.cn` 的 JSON API，主要是 JWT 鉴权、预检接口和下单接口。

把它们放在同一个 Room Bot 里，是因为它们属于同一类问题；把它们分成两个子目录，是因为协议、状态和恢复方式不一样。

## 归档时的取舍

项目进入暂时不用的状态后，归档的目标不是“还能马上冲”，而是“以后能看懂、能恢复、能知道当时为什么这么做”。因此归档时做了几个取舍：

1. 删除二进制和虚拟环境，因为它们可以重建。
2. 删除多端口复制代码，因为它们会制造维护噪音。
3. 保留样例配置，因为它说明字段结构。
4. 本地保留真实配置，但不提交公开仓库。
5. 补全文档，把技术判断和踩坑经验写下来。

这也是把项目从“临时脚本”收束成“可复盘工程”的关键一步。
