# 遇到的困难

项目真正困难的地方不在于发一个 HTTP 请求，而在于请求必须在正确时间、用正确身份、带正确安全字段、穿过正确网络路径，并且失败时能判断是哪一层出了问题。

## 1. Cookie 不是一个字段，而是一组状态

南山项目里，`PHPSESSID` 只代表会话的一部分。实际请求还会受到 `_identity`、`_csrf`、`acw_tc`、`acw_sc__v3`、统计 Cookie 和微信 token 的影响。最早如果只复制单个字段，很容易出现页面能打开但提交失败，或者提交接口直接跳登录页。

最后的处理方式是支持 `raw_cookie` 优先：能从抓包里拿完整 Cookie，就尽量完整保存。只有没有完整 Cookie 时，才用拆分字段拼装。

## 2. CSRF 来源不稳定

提交接口要求 `_csrf`，但 token 可能出现在不同位置：

- 页面 JS 变量。
- HTML meta。
- 隐藏 input。
- JSON 片段。
- `_csrf` Cookie 的序列化内容。

因此系统不能只写一个正则。最终实现是多策略提取：先访问活动页解析页面，页面没有暴露时，再从 Cookie 中反序列化真实 token；同时做短期缓存和过期回退，减少高峰期重复请求活动页。

## 3. WAF 和滑块会改变响应形态

同一个 URL，正常时返回活动页，异常时可能返回 WAF 页面、HTML 登录页、网关错误页或空响应。只看 HTTP 状态码不够，因为有些失败会以 `200` 返回 HTML。

所以系统增加了响应分类：

- HTML 标题识别。
- 登录页识别。
- WAF 标记识别。
- JSON 业务响应识别。
- 成功样式文案识别。

这让日志从“请求失败”变成“更像登录态失效、WAF 拦截、代理异常或业务已满”。

## 4. 高频提交容易自我阻塞

一开始直觉是频率越高越好，但实践里会遇到连接池耗尽、线程阻塞、代理失效、服务端限流、前端 SSE 断开等问题。高频不是目的，有效请求才是目的。

南山项目最后引入了：

- 全局线程池。
- 单账号最大在途请求数。
- 动态请求超时。
- 可恢复状态码重试。
- 手动任务替换和停止信号。
- SSE 心跳和重连。

这些机制让系统在高峰期不至于因为自己的并发把自己拖死。

## 5. 多端口部署带来重复文件

历史上为了多端口运行，复制了 `dist_5001`、`dist_5002` 等多份代码。这种方式短期直观，但长期会让补丁、配置、文档和安全修复分散到多个副本。

归档时改成一份源码、多份配置的思路。端口应该由启动参数、进程管理器或 systemd 决定，而不是复制整套代码决定。

## 6. 公开归档必须处理历史风险

当前文件脱敏还不够。如果旧提交里曾经出现过 token、代理 API、Cookie、身份证等内容，把仓库直接改成 Public 就会把历史也公开。因此发布公开版时，不能只做普通提交，需要用干净历史重新发布，或者彻底清洗历史。

本次归档采用干净公开归档的方向：源码和文档可公开，真实配置留本地，不进入公开历史和 Release 源码包。这个取舍牺牲了旧提交的连续性，但换来的是更清晰的安全边界。
