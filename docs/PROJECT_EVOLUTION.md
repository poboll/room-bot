# 项目全过程复盘

这份文档记录 Room Bot 从临时脚本到公开归档的演进。它不是流水账，而是把每个阶段的技术判断、踩坑和取舍讲清楚。

## 1. 起点：先把人工动作跑通

最早的问题很朴素：开放窗口短，人工操作慢，多账号资料重复录入，登录态和滑块还会打断节奏。因此第一阶段不是先设计系统，而是先确认：

- 平台到底用 Cookie 还是 token。
- 提交按钮背后是哪一个接口。
- 提交字段能不能从抓包复现。
- 成功、已满、登录过期分别长什么样。

这个阶段的产物通常很粗糙：一个能发请求的 Python 脚本，一段硬编码 token 或 Cookie，一组写死的房源 ID。

## 2. 第一版：能请求，但不稳定

早期脚本的目标是“能提交”。它解决了最小闭环：

```text
配置个人资料
  -> 拼请求头
  -> 拼 payload
  -> POST 接口
  -> 打印返回
```

问题也很快暴露：

- Cookie 放在代码里，不安全，也不方便换账号。
- 只看一个返回字段，分不清登录过期和房源已满。
- 高频请求没有边界，容易让自己阻塞。
- 多端口部署复制整份代码，后续补丁难同步。

## 3. 配置化：账号和代码分离

第二阶段把账号信息移到 `config.json`。这样新增账号时，只需要追加配置，不再改代码。

南山账号配置包含：

- 账号名和启用状态。
- `raw_cookie`。
- 姓名、身份证、手机号、学校、专业、毕业日期。
- 户籍类型、入住日期、学信网验证码。
- 意向企业和意向地区。

这个阶段最重要的经验是：真实身份资料只能在本地配置里，不能进入默认代码、README 或公开仓库。

## 4. 抓包深化：从 Cookie 到 CSRF/WAF

一开始以为复制 Cookie 就够了。后来发现南山还需要处理：

- `_csrf` 表单字段。
- `_csrf` Cookie 里的序列化 token。
- 活动页里的 token 提取。
- 阿里云 WAF 和滑块。
- `acw_sc__v3` 的时效性。

于是脚本从“直接提交”变成：

```text
构造 Cookie
  -> 访问活动页
  -> 提取或还原 CSRF
  -> 识别 WAF/登录页
  -> 提交表单
  -> 分类响应
```

这一步是南山项目真正变复杂的分界线。复杂不是因为代码多，而是因为请求前置条件多。

## 5. 响应分类：把失败拆开

如果日志只写“请求失败”，系统就没有调试价值。后来把失败拆成几类：

- Cookie 缺失或过期。
- CSRF 缺失。
- WAF/滑块拦截。
- 代理鉴权失败。
- HTTP 429/405/5xx。
- HTML 登录页或网关页。
- 业务返回已满。
- 成功样式响应。

这让运行时能判断下一步动作：重新抓 Cookie、重新过滑块、刷新代理、降频等待，还是停止账号。

## 6. 并发与调频：从多打请求到有效请求

高峰期最容易犯的错是把“请求次数”当作目标。实际目标应该是“有效请求比例”。

因此后续加入：

- 全局线程池。
- 单账号在途请求上限。
- 动态请求超时。
- 可恢复状态码重试。
- 代理 TTL 和失败退避。
- 手动任务替换。
- SSE 心跳。

定时模式也分成 keepalive、warmup、peak、cooldown。这样做是为了开放前保活，开放后集中提交，峰值后降频观察。

## 7. 多端口部署：从复制代码到一份源码

历史上为了多个端口同时跑，复制了很多 `dist_50xx` 目录。短期能用，但长期有明显问题：

- 每份代码都可能落后。
- 修一个 bug 要同步很多目录。
- 配置和源码混在一起。
- 二进制、压缩包、日志占空间。

归档时把它收束成一份源码、多份配置的思想。端口应该由启动参数或进程管理器决定，不应该由复制代码决定。

## 8. 青年驿站：同类问题，不同协议

青年驿站和南山都属于预约自动化，但协议不同：

- 南山：Cookie + CSRF + WAF + form submit。
- 青年驿站：Bearer token + JSON check/order。

所以它们放在同一个仓库，是因为问题域相同；放在不同目录，是因为技术链路不同。

青年驿站最终保留两个入口：

- `run.py`：终端循环。
- `web_grab.py`：Web 控制台。

## 9. 归档：从可运行到可复盘

当项目暂时不用时，目标从“马上运行”变成“以后看得懂、恢复得起来、知道当时为什么这么写”。

归档动作包括：

- 删除二进制、日志、缓存、虚拟环境。
- 删除多端口重复代码。
- 保留样例配置。
- 本地保留真实配置。
- 编写技术文档。
- 将公开仓库历史清洗为不含真实凭据的干净历史。

这里也有一个教训：清洗公开历史前应该明确告知会覆盖远端分支历史，并先创建私有备份。后来已补做本地 bundle 备份，但这个确认流程以后必须前置。

## 10. 现在的状态

公开仓库保留：

- `南山/` 源码和说明。
- `青年驿站/` 源码和说明。
- `docs/` 复盘文档。
- 样例配置。

本地保留：

- `南山/config.json`
- `南山/JSON/`
- `青年驿站/config.json`
- `private_history_backup_DO_NOT_PUBLISH/`

公开仓库不保留真实 Cookie、JWT、代理接口、通知密钥和个人身份信息。

## 11. 后续如果重启项目

建议先补这些能力：

1. dry-run 模式：只验证登录态、CSRF、代理和 payload，不提交。
2. 响应样本 fixture：把典型 JSON/HTML/WAF 页面存成脱敏测试样本。
3. 配置校验器：启动前检查账号字段、日期、Cookie 关键项。
4. 加密配置：真实配置用本地加密文件或密码管理器导出。
5. 历史管理规范：涉及公开仓库时，先备份、再清洗、再发布。
